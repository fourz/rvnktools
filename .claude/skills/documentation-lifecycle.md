# Documentation Lifecycle Skill

**Purpose**: Documentation management patterns for creation, classification, synchronization, and archival
**Time Savings**: Reduces documentation confusion by 70% (30min → 9min per decision)
**Pattern**: Structured lifecycle from creation to archival with Archon RAG integration

---

## Overview

This skill defines documentation lifecycle patterns for managing technical documentation through its entire lifecycle: classification, creation, synchronization, archival, and maintenance.

**Benefits**:
- Clear document placement decisions (spec vs guide vs api vs fix)
- Automated Archon RAG synchronization workflows
- Temporal document tracking (bug fixes, incidents)
- Systematic archival process
- Reduced duplication and bloat

---

## Document Classification

### Long-Term Documents (Persistent Knowledge)

**Location**: `docs/{spec|guide|api|reference}/`
**Naming**: `feature-name.md` (NO timestamps)
**Lifecycle**: Create once, update in place
**RAG Sync**: ✅ **MUST sync to Archon**
**Archival**: Move entire document only after deprecation

**Categories**:

| Category | Purpose | Sync? | Examples |
|----------|---------|-------|----------|
| **spec/** | System specifications, architecture | ✅ YES | archon-board-inventory.md, component-inventory.md |
| **guide/** | Tutorials, setup, how-tos | ✅ YES | getting-started.md, installation.md |
| **api/** | API documentation, endpoints | ✅ YES | endpoints.md, authentication.md |
| **reference/** | Configuration, CLI, operational guides, procedures | ✅ YES | configuration.md, environment-variables.md, archon-backup-strategy.md |

**Workflow**:
```
1. Create in docs/{category}/feature-name.md
2. Immediately sync to Archon RAG
3. Update in place if changes needed
4. Re-sync after updates
5. Archive entire doc (if deprecated)
```

### Short-Term Documents (Temporal Work)

**Location**: `docs/fix/`
**Naming**: `YYYY-MM-DD-issue-name.md` (WITH timestamps)
**Lifecycle**: Created for specific issue, archived after resolution
**RAG Sync**: ❌ **DO NOT sync to Archon**
**Task Reference**: Always link in Archon task
**Archival**: Move to `docs/archive/` when issue resolved

**Purpose**:
- Bug fix documentation
- Incident reports
- Temporary solutions
- Investigation notes
- Short-term decisions

**Workflow**:
```
1. Create with timestamp: docs/fix/YYYY-MM-DD-issue.md
2. Link in Archon task: /docs/fix/YYYY-MM-DD-issue.md
3. DO NOT sync to RAG (temporal only)
4. After resolution → move to docs/archive/
5. Task completed → mark as "done"
```

**Examples**:
```
docs/fix/2025-12-22-connection-timeout.md
docs/fix/2025-12-18-auth-error-handling.md
docs/fix/2025-12-10-performance-regression.md
```

### Special Purpose Documents

#### Release Documentation

**Location**: `docs/release/`
**Naming**: Version-based (`changelog.md`, `v1-0-0-release-notes.md`) or migration-based (`migration-to-v2.md`)
**Lifecycle**: Persistent per version, archive old releases
**RAG Sync**: ⚠️ Limited (version-specific docs, archive old versions)
**Purpose**: Release notes, changelog entries, breaking changes, deployment procedures for Ravenkraft projects

#### Milestone Reports

**Location**: `docs/milestones/`
**Naming**: `milestone-name.md` (e.g., `language-enumeration-complete.md`)
**Lifecycle**: Point-in-time report, archive after project milestone
**RAG Sync**: ❌ **DO NOT sync** (historical reports, not persistent knowledge)
**Purpose**: Project milestone completion reports, typically verbose documentation generated by Sonnet
**Archival Criteria**: Archive milestones >3 months old or upon next major milestone completion

#### Operations/Procedures (Merged into Reference)

**Location**: `docs/reference/`
**Naming**: `operation-name.md` (e.g., `archon-backup-strategy.md`)
**Lifecycle**: Persistent, update as procedures change
**RAG Sync**: ✅ **YES** (operational guides are reference knowledge)
**Purpose**: Operational procedures, backup strategies, maintenance guides
**Note**: `docs/operations/` contents should be moved to `docs/reference/`

### Archival Documents

**Location**: `docs/archive/`
**Naming**: Preserved from original location
**Retention**: Searchable, not active
**RAG Sync**: ❌ **NOT indexed** (remove from RAG if previously synced)

**Migration Path**:
```
docs/spec/deprecated-feature.md       → docs/archive/deprecated-feature.md
docs/fix/2025-11-20-old-issue.md      → docs/archive/2025-11-20-old-issue.md
docs/milestones/old-milestone.md      → docs/archive/2025-01-old-milestone.md
```

## Document Creation Decision Tree

```
┌─ Is this maintained knowledge or specification?
│  ├─ YES → Long-Term
│  │   ├─ Is it about system design/architecture?
│  │   │  └─ YES → docs/spec/feature-name.md
│  │   ├─ Is it a tutorial or how-to?
│  │   │  └─ YES → docs/guide/feature-name.md
│  │   ├─ Is it API documentation?
│  │   │  └─ YES → docs/api/feature-name.md
│  │   └─ Is it a reference document?
│  │      └─ YES → docs/reference/feature-name.md
│  │
│  └─ NO → Is this a bug fix or temporal work?
│     ├─ YES → Short-Term
│     │  └─ docs/fix/YYYY-MM-DD-issue.md
│     └─ NO → Unclear. Ask: Is this maintainable long-term?
│        ├─ YES → Long-Term category (choose above)
│        └─ NO → Short-Term (with timeline)
```

## Naming Standards

### Long-Term Documents (NO Timestamps)

**Format**: `kebab-case-descriptive-name.md`

**Examples**:
```
✅ documentation-lifecycle-workflow.md
✅ archon-board-inventory.md
✅ authentication-guide.md
✅ api-reference.md

❌ 2025-12-22-feature.md (timestamp not allowed)
❌ My_Feature_Doc.md (underscores not allowed)
❌ feature123.md (not descriptive)
```

### Short-Term Documents (WITH Timestamps)

**Format**: `YYYY-MM-DD-issue-description.md`

**Examples**:
```
✅ 2025-12-22-connection-timeout.md
✅ 2025-12-18-auth-error-handling.md
✅ 2025-12-10-performance-regression.md

❌ connection-timeout.md (missing timestamp)
❌ 12-22-2025-issue.md (wrong timestamp format)
```

### Archive Migration

**Preserve original naming**:
```
docs/spec/old-feature.md → docs/archive/old-feature.md
docs/fix/2025-09-15-bug.md → docs/archive/2025-09-15-bug.md
```

**Update ARCHIVE_MANIFEST.md**:
```markdown
- **old-feature.md**: Deprecated and replaced by new-feature.md (2025-12-22)
- **2025-09-15-bug.md**: Issue resolved, archived (2025-12-20)
```

## Archon RAG Synchronization

### Sync Types

| Type | Trigger | Frequency | Tool | Output |
|------|---------|-----------|------|--------|
| **Immediate** | New long-term doc | On creation | `/archon-sync` | Indexed in 5-10 min |
| **Batch** | Multiple docs | Weekly/ad-hoc | `/doc-batch-sync` | Batch report + validation |
| **Scheduled** | Recurring | Weekly | Claude Hooks | Automated |
| **Manual** | Edge cases | As needed | CLI command | Custom validation |

### Immediate Sync Pattern

**When**: Just created a long-term document

```bash
# Create document
echo "# Feature Name" > docs/spec/feature-name.md

# Sync immediately
claude /archon-sync full
```

**Expected Output**:
```
✅ Found documentation directory
✅ Converted markdown → JSON
✅ Updated Archon (1 doc)
✅ RAG indexing: Check in 5-10 minutes
```

### Batch Sync Pattern

**When**: Creating/updating multiple docs (3+)

```bash
# Sync all docs/
claude /doc-batch-sync docs sync

# Sync specific category
claude /doc-batch-sync spec sync

# Preview only (dry run)
claude /doc-batch-sync docs dryrun
```

### Sync Validation

**Wait 5-10 minutes** for RAG indexing, then test:

```javascript
// Verify sync successful
rag_search_knowledge_base(query="feature name", match_count=3)
rag_search_knowledge_base(query="specific content", match_count=3)
```

**Expected**: Queries return relevant results from synced docs

**If validation fails**:
1. Wait 10 more minutes (indexing delay)
2. Check Archon health: `mcp__archon__health_check()`
3. Retry sync: `claude /archon-sync full`
4. Review Archon instance for errors

## Archival Workflow

### Archival Criteria

**Archive when**:
- ✅ Document is deprecated/obsolete
- ✅ Content moved to other doc (consolidation)
- ✅ >6 months old AND not actively maintained
- ✅ Replaced by newer specification

**DO NOT archive**:
- ❌ Current specifications (spec/)
- ❌ Active tutorials (guide/)
- ❌ Current API documentation (api/)
- ❌ Configuration reference (reference/)

### Archival Process

**Step 1**: Identify candidates
```bash
# Find docs older than 6 months in docs/fix/
find docs/fix -name "*.md" -mtime +180

# Check for deprecation markers in spec/
grep -r "DEPRECATED" docs/spec/
```

**Step 2**: Move to archive (preserve git history)
```bash
# Move with git to preserve history
git mv docs/spec/deprecated-feature.md docs/archive/deprecated-feature.md

# Update ARCHIVE_MANIFEST.md
echo "- **deprecated-feature.md**: Replaced by new-feature.md ($(date +%Y-%m-%d))" >> docs/archive/ARCHIVE_MANIFEST.md
```

**Step 3**: Update cross-references
```bash
# Find references to archived doc
grep -r "deprecated-feature.md" docs/ .claude/

# Update found references to point to new location or new doc
```

**Step 4**: Remove from Archon RAG (if synced)
```bash
# Re-sync to update Archon knowledge base
claude /archon-sync full
```

## Health Metrics

### Documentation Health Indicators

| Metric | Healthy | Warning | Critical |
|--------|---------|---------|----------|
| **Long-term docs synced** | >95% | 80-95% | <80% |
| **Fix docs >90 days old** | <5 | 5-10 | >10 |
| **Broken cross-references** | 0 | 1-3 | >3 |
| **Archon sync lag** | <10 min | 10-30 min | >30 min |

## Best Practices

✅ **DO**:
- Sync long-term docs immediately after creation
- Use timestamps for temporal docs (docs/fix/)
- Validate RAG queries after sync (wait 5-10 min)
- Archive temporal docs after resolution
- Preserve git history when moving files (git mv)
- Update ARCHIVE_MANIFEST.md when archiving
- Use kebab-case for all filenames
- Choose correct category (spec/guide/api/reference)

❌ **DON'T**:
- Sync temporal docs to Archon RAG (docs/fix/)
- Use timestamps for long-term docs
- Move files without git mv (loses history)
- Archive active specifications
- Leave orphaned cross-references after moves
- Skip Archon sync for long-term docs
- Create docs without category (root-level)
- Use underscores or spaces in filenames

## Common Workflows

### Creating Long-Term Specification

```bash
# 1. Create document
cat > docs/spec/new-feature.md << 'EOF'
# New Feature Specification
...content...
EOF

# 2. Sync immediately
claude /archon-sync full

# 3. Validate (wait 5-10 min)
rag_search_knowledge_base(query="new feature", match_count=3)
```

### Creating Temporal Bug Fix Doc

```bash
# 1. Create with timestamp
cat > docs/fix/2025-12-22-auth-bug.md << 'EOF'
# Issue: Authentication Bug
**Task**: [link to Archon task]
...investigation...
EOF

# 2. Link in Archon task
manage_task("create", project_id="...",
  title="Fix auth bug",
  description="See /docs/fix/2025-12-22-auth-bug.md")

# 3. After resolution, archive
git mv docs/fix/2025-12-22-auth-bug.md docs/archive/
echo "- **2025-12-22-auth-bug.md**: Resolved (2025-12-23)" >> docs/archive/ARCHIVE_MANIFEST.md
```

### Monthly Cleanup Workflow

```bash
# 1. Audit current state
claude /doc-cleanup audit

# 2. Archive old temporal docs (>90 days)
find docs/fix -name "*.md" -mtime +90 -exec git mv {} docs/archive/ \;

# 3. Update ARCHIVE_MANIFEST.md
# (add entries for each archived doc)

# 4. Re-sync to update Archon
claude /archon-sync full

# 5. Validate
claude /doc-cleanup validate
```
